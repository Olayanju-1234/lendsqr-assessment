1. Overview

Demo Credit is a wallet service MVP for a mobile lending application. It enables users to create accounts, fund their wallets, transfer money to other users, and withdraw funds. The system integrates with Lendsqr's Adjutor Karma API to prevent blacklisted users from being onboarded.

3. Architecture

I followed a 4-layer architecture pattern:

Routes → Controllers → Services → Models

  • Routes map HTTP paths to controllers and apply validation middleware
  • Controllers extract request data and format responses — they contain zero business logic
  • Services handle all business logic, transaction scoping, and validation rules
  • Models are thin data access wrappers around KnexJS queries

This separation ensures each layer has a single responsibility. Dependencies flow inward — services never import Express, controllers never touch the database directly. This makes the codebase testable, maintainable, and easy to reason about.

4. Key Design Decisions

4.1 Double-Entry Bookkeeping for Transfers

Every transfer creates two transaction records: a debit for the sender and a credit for the recipient. Both records share the same UUID reference, linking them together. This is standard in financial systems because it ensures balances always reconcile and provides a complete audit trail from both perspectives.

4.2 Pessimistic Locking with Deadlock Prevention

All wallet mutations use SELECT ... FOR UPDATE inside MySQL transactions to prevent race conditions. For transfers involving two wallets, I lock them in ascending user_id order to prevent deadlocks. This consistent lock ordering guarantees that two simultaneous transfers between the same users will never deadlock each other.

4.3 Idempotency Key Support

All wallet operations (fund, transfer, withdraw) accept an optional Idempotency-Key header. If a client retries a request with the same key, the system returns the cached result instead of processing a duplicate transaction. This prevents double-charges caused by network retries or client-side failures — a critical requirement for any financial API.

4.4 Fail-Open Strategy for Adjutor API

If the Adjutor Karma API is unavailable or times out (5-second limit), registration proceeds rather than blocking all signups. This is a deliberate trade-off: a brief API outage should not prevent legitimate users from onboarding. The risk of a blacklisted user slipping through during downtime is acceptable compared to blocking all registrations entirely.

4.5 Atomic User + Wallet Creation

User and wallet records are created in a single database transaction during registration. This prevents orphaned users (users without wallets) that would cause errors during wallet operations.

4.6 Precision-Safe Financial Arithmetic

All money fields use MySQL's DECIMAL(15,2) type. JavaScript arithmetic results are rounded to 2 decimal places using parseFloat(result.toFixed(2)) to prevent floating-point precision errors that could cause balance drift over time.

4.7 Immutable Audit Trail

Transaction records are append-only. Each record captures balance_before and balance_after, enabling full reconstruction of any wallet's history. No transaction record is ever updated or deleted.

5. Security Measures

  • Helmet for HTTP security headers (X-Content-Type-Options, X-Frame-Options, etc.)
  • Rate limiting — 100 requests per 15 minutes globally, 20 per 15 minutes on auth endpoints to prevent brute-force attacks
  • CORS middleware for cross-origin request handling
  • Bcrypt password hashing with 10 salt rounds
  • Joi validation on all request inputs with strict schemas (positive amounts, max 2 decimal places, upper limits)
  • KnexJS parameterized queries — prevents SQL injection
  • JSON payload size limit (10KB) to prevent large payload attacks
  • Passwords never returned in any API response
  • Morgan request logging for monitoring and debugging

8. Testing Strategy

24 unit tests across 3 test suites covering both positive and negative scenarios:

  • Auth Service (7 tests): Registration (valid, blacklisted user, duplicate email, API down), Login (valid, wrong password, non-existent email)
  • Wallet Service (14 tests): Fund, transfer (with dual-entry verification and lock ordering), withdraw, balance, idempotency (cached result, new key) — plus edge cases for insufficient funds, self-transfer, missing wallets
  • Transaction Service (3 tests): Paginated history, page calculation, empty results

All tests mock the database layer, so no MySQL instance is required to run the test suite.
